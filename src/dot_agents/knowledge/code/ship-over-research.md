# Ship Over Research

> Define success criteria up-front and ship minimal scope — doing teaches more than researching.

## Key Takeaways

- Two project trajectories: quick execution that ships, or extended research that stalls. Output beats consideration — "I'd much rather have *done* a lot than have only *considered* a lot."
- Write the success criteria *before* starting. A vague goal invites endless feature additions; a concrete goal ("jam on woodworking with a friend") tells you when to stop.
- Powerful tools tempt scope creep. Discovering a capable library invites implementing features you'll never use. Ask whether each addition serves the actual use case.
- Faster programming doesn't shorten projects: "Any increases in programming speed will be offset by a corresponding increase in unnecessary features, rabbit holes, and diversions."
- Long-term research interests stall in research mode. Concrete projects with crisp success criteria advance; open-ended explorations don't.
- Even projects that "obviously" fail teach more than hypothetical planning. Ship to learn.

# Code Implement Spec

Implement tasks from a reviewed plan. This is adapted from Cavekit's `build`
skill, with this repo's dated plan convention: plans usually live at
`docs/plans/YYYY-MM-DD-<slug>.md`, not root `SPEC.md`.

## Load

1. Find the target plan:
   - If the user names a file, use it.
   - Otherwise search `docs/plans/` for the most relevant dated plan.
   - If multiple plausible plans exist, ask which one to use.
   - If no plan exists, stop and ask whether to run `code-generate-spec`.
2. Read the full plan.
3. If the plan uses compact `§` sections, read
   `../code-generate-spec/references/FORMAT.md` for table/status/symbol rules.
4. Parse requested task scope:
   - `§T.n` or `Tn`: that task only.
   - `--next` or "next": lowest-numbered `§T` row with status `.` or `~`.
   - `--all` or no task selector: every `.` row in `§T` order.
5. If there is no `§T` table, infer the smallest actionable task list from the
   plan and ask before proceeding.

## Plan Before Editing

Before code edits for the selected task(s), produce a short implementation plan:

- Task IDs and current statuses.
- `§V` invariants that must hold.
- `§I` interfaces touched.
- Files to create or edit.
- Tests to add or update, one per touched invariant when possible.
- Verification command(s), including final full-suite command when known.

If the user explicitly asked to implement now, continue after presenting this
plan. Otherwise ask for approval before editing.

## Execute

For each selected task, in order:

1. Flip the task status in the plan from `.` to `~`.
2. Edit code only within the selected task scope.
3. Add or update tests promised in the implementation plan.
4. Run the task verification command.
5. If verification passes, flip the task status from `~` to `x`.
6. Continue to the next selected task.

At the end, run the broadest practical validation command from the plan or repo.

## Failure Handling

When verification fails, do not retry blindly.

1. Read the failure output and identify the root cause.
2. Classify it:
   - Code bug: fix code, rerun verification. No plan change except task status.
   - Plan bug: use `code-generate-spec` AMEND/BACKPROP rules to update the plan,
     then resume against the updated plan.
   - Unspecified edge case: use `code-generate-spec` BACKPROP rules to add a
     `§B` bug row, a `§V` invariant when useful, and matching `§T`/`§Q`/`§F`
     coverage, then resume.
3. Never silently fix a root cause that should become a plan invariant.

## Write Policy

- This skill may only update `§T` status cells directly.
- Any other plan edit must follow `code-generate-spec` AMEND or BACKPROP.
- Do not perform speculative work outside the selected task scope.
- Do not create progress dashboards; `§T` is the dashboard.
- Commit only when the user asked for commits or the surrounding workflow
  explicitly requires them.

## Done Criteria

A task is `x` only when:

- Its verification command exits 0.
- Required tests were added or updated.
- Touched `§V` invariants still hold.
- No selected task remains `~`.

Report the final task statuses, validation commands, and any plan backprops.
